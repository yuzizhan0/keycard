# Keycard — 本地 API Key 管理（设计规格）

**日期:** 2026-03-29  
**状态:** 已定稿（用户整体验收 §1–§4）；经规格审阅修订

## 背景与目标

用户在各 AI Agent / IDE / CLI 场景下持有大量模型与服务的 API key，分散且敏感。Keycard 是一个**本地优先、开源**的工具：在浏览器中复制 key 后可通过快捷方式入库；在终端与桌面配置场景中通过 profile 安全消费。首版不依赖浏览器扩展；扩展作为二期能力。

## §1 架构与组件

### `keycard-core`（Rust 库，同仓）

- 负责加密保管、SQLite 持久化、领域模型（条目、profile、设置）。
- **Tauri 应用与 CLI 仅通过此库访问存储**，禁止重复实现存储或加密逻辑。

### Tauri 桌面应用

- **托盘常驻**；注册**全局快捷键**。
- **「快速保存」小窗**：将剪贴板内容作为密钥值的默认来源，用户补全别名、提供商等后写入库。
- **主窗口**：检索、编辑元数据、与复制相关的安全行为（如可选复制后清空剪贴板）、会话锁定。

### CLI `keycard`

- 与桌面应用使用**同一数据目录与同一数据库文件**。
- 负责终端侧 **profile → 子进程环境变量注入**（见 §3）。

### 二期：浏览器扩展

- 仅负责将 **URL、页面标题等上下文** 传递给本机应用；**密钥不进入扩展的持久化存储**。
- 通信通道在实现前在规格/实现计划中**二选一写死**：**Native Messaging** 或 **本机 loopback + 一次性 token**，并配套威胁模型说明（何种主体可以连接、如何鉴权）。

## §2 数据与加密

### 存储

- 使用**单个 SQLite** 数据库文件；默认**文件名**为 `vault.db`，位于各平台**应用数据目录**下的 `Keycard/` 子目录（便于用户备份与排错）：
  - **macOS:** `~/Library/Application Support/Keycard/vault.db`
  - **Linux:** `$XDG_DATA_HOME/Keycard/vault.db`；若未设置 `XDG_DATA_HOME`，则为 `~/.local/share/Keycard/vault.db`
  - **Windows:** `%LOCALAPPDATA%\Keycard\vault.db`（通常为 `C:\Users\<用户>\AppData\Local\Keycard\vault.db`）

### 并发与数据库

- 连接使用 **SQLite WAL** 与**合理 busy timeout**（实现时默认建议 ≥ 5000 ms），以允许 **GUI 与 CLI 短事务交错**。
- **预期用法**：读写均为短事务；避免长时间占用写锁。若发生 `SQLITE_BUSY`，应向用户给出可重试的说明。
- **策略**：所有数据库访问**仅通过 `keycard-core`**，便于统一 journal 模式与超时配置。

### 锁定后行为

- **桌面已锁定**时：**快速保存**与**需解密的写操作**应先要求解锁（或明确阻止并提示），不得在无解锁状态下写入 secret。
- **主窗口**显示锁定状态；已解锁前不展示明文 secret（若 UI 需展示，仅显示掩码或留空）。
- **CLI**：首版按下方「CLI 与解锁」**必须采用 (a)**，锁定桌面与否不影响 CLI——CLI **独立提示主密码**（与桌面会话无 IPC 共享）。

### 主密码与密钥派生

- 首次创建库时设置**主密码**；使用 **Argon2id** 派生密钥。
- **主密钥不以明文形式写入磁盘**。
- 支持**可配置的空闲自动锁定**；锁定后的交互规则见上文「锁定后行为」。

### 密码学规范（首版须遵循）

- **口令派生：** **Argon2id**，参数目标为 OWASP 当前建议量级（实现时固定写入代码与 README）：**m ≥ 64 MiB**、**t ≥ 3**、**p = 4**（若将来调整须 bump schema / 迁移策略并文档说明）。
- **对称加密：** **AEAD** 选用 **XChaCha20-Poly1305** 或 **AES-256-GCM** 之一（实现时全仓统一）；**每条 secret 使用独立随机 nonce**（长度符合所选算法要求），**禁止**在同一数据密钥下复用 nonce。
- **密钥层次：** 由主密码派生出的**主密钥**经 **HKDF**（或等价 KDF）派生**数据加密密钥**与**其他用途子密钥**（如需要），避免同一原始密钥多用途混用；具体标签在实现文档中列出。

### 密钥与元数据

- **每条 secret 使用独立 AEAD**；库内仅存密文与 nonce。
- **元数据**（提供商、别名、标签、创建时间等）可**明文**存储于 SQLite，以支持检索与展示。
- **日志与错误信息中不得包含 secret 明文。**

### CLI 与解锁

- **首版必须实现 (a)，且不得以 (b) 作为首版默认或并存路径：** CLI 在需要访问密钥的子命令中**提示输入主密码**（或等价 tty 解锁），与桌面是否已解锁**无关**。
- **(b)**（桌面解锁后通过本机 IPC 授权 CLI）仅可作为**明确标记为实验性**的后续迭代，需单独评审威胁模型后再引入。

### 同步

- **首版不做云同步**。备份方式为用户自行拷贝加密后的数据库文件；若未来增加同步，需单独设计与评审（端到端加密同步等）。

## §3 入库与出库

### 入库（浏览器 copy 后）

- 通过**全局快捷键**或**托盘菜单**打开快速保存小窗。
- **密钥值默认来自剪贴板**，用户可编辑。
- **别名必填**；提供商为**预设列表 + 自定义**。
- **保存成功后是否清空剪贴板**：独立选项；**默认建议关闭**，若开启需在 UI 中给出风险提示。
- **形态检测**（如 `sk-` 前缀、`Bearer` 等）仅作 **UI 提示**，不自动改写剪贴板或密钥内容。

### 出库（终端 / 桌面消费）

- **Profile**：命名的一组映射：**环境变量名 → 库内条目 id**。
- CLI 能力（具体 flag 名称在实现前统一）：
  - `keycard run --profile <name> -- <command>`：为子进程注入对应环境变量后执行命令。
  - `eval "$(keycard env --profile <name>)"`：在当前 shell 导出变量（需在文档中说明与 shell 历史相关的风险）。
- 文档中强调：优先使用 **`run` 子进程注入**，**尽量避免**将 `export KEY=明文` 写入 shell 历史。

## §4 错误处理、安全与测试

### 错误处理

- **主密码错误**、**库损坏或 schema 版本不兼容**、**全局快捷键注册失败**等：提供用户可读的说明；CLI 以**非零退出码**表示失败。
- 任何日志、诊断输出**不得包含 secret**。

### 安全与信任

- 在文档中说明**威胁模型**：本机恶意软件、备份介质泄露、多人共用机器等场景下的风险与建议。
- 依赖版本与**可复现构建**在 README / CONTRIBUTING 中说明，便于社区审计。

### 测试

- **`keycard-core`**：单元测试覆盖加解密、读写、profile 解析等核心逻辑。
- **CLI**：在临时目录下进行集成测试。
- **全局快捷键与各操作系统差异**：发布前使用**手动测试清单**验证。

## 范围外（首版明确不做）

- 云同步与多设备自动同步。
- 浏览器扩展（二期）。
- 作为通用密码管理器替代 1Password / Bitwarden 等全功能竞品。

## 首版验收标准（Definition of Done）

以下项可独立验证，作为 v1 完成的最低标准：

1. **建库与解锁：** 可创建加密库、用主密码解锁桌面应用；空闲锁定与锁定后行为符合「锁定后行为」。
2. **快速保存：** 全局快捷键与托盘均可打开快速保存；剪贴板预填、别名必填、可选保存后清剪贴板与形态提示符合 §3。
3. **主窗口：** 可列表、搜索、编辑元数据；复制 secret 行为符合设置；锁定时不泄露明文。
4. **CLI：** `keycard run` / `keycard env`（或最终统一的子命令名）在**提示主密码**后可按 profile 注入环境变量；失败时非零退出。
5. **存储路径：** 默认 `vault.db` 路径符合 §2「存储」中的平台约定，并在 README 中再次写明。
6. **数据库：** 启用 WAL 与 busy 策略，GUI+CLI 短事务场景下无未文档化的破坏性冲突。
7. **日志与错误：** 自动化或手动检查典型路径下日志/错误输出**不含** secret 明文。
8. **测试：** `keycard-core` 加解密与 profile 单测通过；CLI 在临时目录的集成测试通过。
